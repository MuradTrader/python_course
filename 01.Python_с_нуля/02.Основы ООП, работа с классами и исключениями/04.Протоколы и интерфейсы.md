## Занятие 13: Протоколы и интерфейсы в Python

Сегодня разберём, как в Python создаются **протоколы** (интерфейсы) с помощью **абстрактных базовых классов (ABC)** и множественного наследования.
Протокол определяет, какие методы и атрибуты должен иметь объект, чтобы его можно было использовать в определённом контексте.
В Python нет встроенного ключевого слова `interface`, как в Java или Go, но мы можем эмулировать интерфейсы через наследование от `ABC` и декоратор `@abstractmethod`.

---

## 1. Создаём абстрактный базовый класс (протокол) `Switchable`

Создадим файл `switchable.py` (в пакете `switchables`). В нём опишем класс `Switchable`, который будет нашим интерфейсом для всех "включаемых/выключаемых" устройств.

```python
# switchables/switchable.py
from abc import ABC, abstractmethod

class Switchable(ABC):
    @abstractmethod
    def turn_on(self):
        """Включить устройство"""
        pass

    @abstractmethod
    def turn_off(self):
        """Выключить устройство"""
        pass
```

**Объяснение:**

- `ABC` – это класс из модуля `abc` (abstract base class). Он использует метакласс `ABCMeta`, который позволяет помечать методы как абстрактные.
- `@abstractmethod` – декоратор, который говорит: «этот метод **обязаны** переопределить все неабстрактные наследники».
- Класс, у которого есть хотя бы один абстрактный метод, **нельзя инстанциировать** (создать экземпляр). Это и есть суть интерфейса – он задаёт контракт, но не даёт готовой реализации.

## 2. Пробуем создать экземпляр `Switchable`

Создадим главный файл `main.py` и попробуем напрямую создать объект `Switchable`.

```python
# main.py
from switchables.switchable import Switchable

def main():
    s = Switchable()   # попытка создать экземпляр
    print(s)

if __name__ == "__main__":
    main()
```

**Вывод в консоли:**

```
TypeError: Can't instantiate abstract class Switchable with abstract methods turn_off, turn_on
```

Python не позволяет создать экземпляр, потому что не реализованы абстрактные методы `turn_on` и `turn_off`.
Это и есть защита: интерфейс нельзя использовать напрямую, только через классы, которые выполнят контракт.

## 3. Создаём конкретное устройство – `LightBulb` (лампочка)

Создадим файл `light_bulb.py` в том же пакете `switchables`.

```python
# switchables/light_bulb.py
from .switchable import Switchable

class LightBulb(Switchable):
    def turn_on(self):
        print(f"{self.__class__.__name__} turned on")

    def turn_off(self):
        print(f"{self.__class__.__name__} turned off")
```

Теперь класс `LightBulb` **реализует** оба абстрактных метода.
Мы можем создать его экземпляр.

Добавим в `main.py`:

```python
# main.py (дополняем)
from switchables.light_bulb import LightBulb

def main():
    bulb = LightBulb()
    print(bulb)
    bulb.turn_on()
    bulb.turn_off()

if __name__ == "__main__":
    main()
```

**Вывод в консоли:**

```
<__main__.LightBulb object at 0x...>
LightBulb turned on
LightBulb turned off
```

Всё работает. Мы использовали интерфейс `Switchable` для того, чтобы гарантировать наличие методов `turn_on` и `turn_off`.

## 4. Создаём выключатель `PowerSwitch`, который управляет любым `Switchable`

Теперь сделаем класс `PowerSwitch` – он будет принимать в конструктор устройство, поддерживающее `Switchable`, и переключать его состояние (вкл/выкл).
Создадим файл `power_switch.py` в пакете `switchables`.

```python
# switchables/power_switch.py
from .switchable import Switchable

class PowerSwitch:
    def __init__(self, client: Switchable):
        # Проверка типа (необязательно, но хороший тон)
        if not isinstance(client, Switchable):
            raise TypeError("client must implement Switchable protocol")
        self.client = client
        self._state = False   # начальное состояние выключателя: выключен

    def toggle(self):
        if self._state:
            self.client.turn_off()
            self._state = False
        else:
            self.client.turn_on()
            self._state = True
```

Пояснения:

- `client: Switchable` – аннотация типа (подсказка для IDE и разработчиков).
- `isinstance(client, Switchable)` – настоящая проверка во время выполнения. Если передан объект, не реализующий `Switchable`, будет ошибка.
- `self._state` – хранит текущее состояние **выключателя** (не путать с состоянием лампочки). Когда выключатель включён (`_state == True`), он посылает команду `turn_on`, и наоборот.

Теперь используем выключатель в `main.py`:

```python
# main.py (дополняем)
from switchables.light_bulb import LightBulb
from switchables.power_switch import PowerSwitch

def main():
    bulb = LightBulb()
    switch = PowerSwitch(bulb)

    print("Первый toggle:")
    switch.toggle()   # включит лампочку
    print("Второй toggle:")
    switch.toggle()   # выключит лампочку
    print("Третий toggle:")
    switch.toggle()   # опять включит

if __name__ == "__main__":
    main()
```

**Вывод в консоли:**

```
Первый toggle:
LightBulb turned on
Второй toggle:
LightBulb turned off
Третий toggle:
LightBulb turned on
```

Выключатель работает с любым объектом, который реализует `Switchable`. Это и есть мощь протоколов – мы можем создавать разные устройства (лампочки, радио, обогреватели) и управлять ими одним и тем же выключателем.

## 5. Создаём второй протокол – для самих выключателей (SwitchBase)

Теперь автор хочет определить интерфейс для выключателей. У выключателя должны быть методы `toggle`, `is_turned_on`, а также `is_turned_off` (который можно реализовать через `is_turned_on`).
Создадим файл `switch_base.py` (например, в пакете `interfaces` или прямо в `switchables`).

```python
# switchables/switch_base.py
from abc import ABC, abstractmethod

class SwitchBase(ABC):
    @abstractmethod
    def toggle(self):
        """Переключить состояние"""
        pass

    @abstractmethod
    def is_turned_on(self) -> bool:
        """Возвращает True, если выключатель во включённом состоянии"""
        pass

    def is_turned_off(self) -> bool:
        """Возвращает True, если выключатель выключен"""
        return not self.is_turned_on()
```

Обратите внимание: `is_turned_off` – **конкретный метод**, он не абстрактный. Он использует абстрактный метод `is_turned_on`.
Наследникам нужно будет реализовать только `toggle` и `is_turned_on`, а `is_turned_off` достанется бесплатно.

Теперь заставим наш `PowerSwitch` реализовать этот интерфейс:

```python
# switchables/power_switch.py (обновлённая версия)
from .switchable import Switchable
from .switch_base import SwitchBase

class PowerSwitch(SwitchBase):
    def __init__(self, client: Switchable):
        if not isinstance(client, Switchable):
            raise TypeError("client must implement Switchable protocol")
        self.client = client
        self._state = False

    def toggle(self):
        if self._state:
            self.client.turn_off()
            self._state = False
        else:
            self.client.turn_on()
            self._state = True

    def is_turned_on(self) -> bool:
        return self._state
```

Теперь `PowerSwitch` удовлетворяет протоколу `SwitchBase`. Проверим в `main.py`:

```python
# main.py (дополняем)
from switchables.light_bulb import LightBulb
from switchables.power_switch import PowerSwitch

def main():
    bulb = LightBulb()
    switch = PowerSwitch(bulb)

    print(f"Initially: turned_on = {switch.is_turned_on()}, turned_off = {switch.is_turned_off()}")
    switch.toggle()
    print(f"After toggle: turned_on = {switch.is_turned_on()}, turned_off = {switch.is_turned_off()}")
    switch.toggle()
    print(f"After another toggle: turned_on = {switch.is_turned_on()}, turned_off = {switch.is_turned_off()}")

if __name__ == "__main__":
    main()
```

**Вывод в консоли:**

```
Initially: turned_on = False, turned_off = True
LightBulb turned on
After toggle: turned_on = True, turned_off = False
LightBulb turned off
After another toggle: turned_on = False, turned_off = True
```

Всё работает как ожидалось.

## 6. Использование `property` для удобства

Автор упоминает, что можно сделать `is_turned_on` свойством (`property`). Это позволяет писать `switch.is_turned_on` без скобок.
Перепишем `SwitchBase` с использованием `@property`:

```python
# switchables/switch_base.py (вариант с property)
from abc import ABC, abstractmethod

class SwitchBase(ABC):
    @abstractmethod
    def toggle(self):
        pass

    @property
    @abstractmethod
    def is_turned_on(self) -> bool:
        pass

    @property
    def is_turned_off(self) -> bool:
        return not self.is_turned_on
```

В классе `PowerSwitch` нужно будет определить `is_turned_on` как свойство:

```python
# switchables/power_switch.py (адаптация)
class PowerSwitch(SwitchBase):
    # ... __init__ и toggle без изменений ...

    @property
    def is_turned_on(self) -> bool:
        return self._state
```

Тогда в `main.py` можно писать:

```python
print(f"turned_on = {switch.is_turned_on}")   # без скобок
```

Вывод будет тот же, но код выглядит чище.

## 7. Добавляем ещё одно устройство – `Radio`

Покажем, что протокол `Switchable` может быть реализован совершенно разными классами.
Создадим `radio.py` в пакете `switchables`:

```python
# switchables/radio.py
from .switchable import Switchable

class Radio(Switchable):
    def turn_on(self):
        print("Radio: playing music")

    def turn_off(self):
        print("Radio: silent")
```

Теперь в `main.py` используем `Radio` с тем же `PowerSwitch`:

```python
# main.py
from switchables.radio import Radio
from switchables.power_switch import PowerSwitch

def main():
    radio = Radio()
    switch = PowerSwitch(radio)

    switch.toggle()   # включить радио
    switch.toggle()   # выключить радио

if __name__ == "__main__":
    main()
```

**Вывод:**

```
Radio: playing music
Radio: silent
```

Мы не меняли `PowerSwitch` – он работает с любым `Switchable`. Это и есть суть интерфейсов: код становится гибким и переиспользуемым.

## 8. Почему это важно? Преимущества протоколов/интерфейсов

- **Контракт** – вы точно знаете, какие методы есть у объекта.
- **Полиморфизм** – один и тот же код (например, `PowerSwitch`) работает с разными типами устройств.
- **Раннее обнаружение ошибок** – если класс не реализует абстрактный метод, вы получите ошибку уже при попытке создать экземпляр (или при наследовании, если класс тоже абстрактный).
- **Документация** – интерфейс сам по себе описывает ожидаемое поведение.
- **Поддержка IDE** – автодополнение и подсказки работают лучше.

## 9. Множественное наследование и примеси (mixins)

В Python класс может наследоваться от нескольких базовых классов. Это позволяет комбинировать разные протоколы.
Например, если у вас есть интерфейс `VolumeControl` (громкость) и интерфейс `Switchable`, то класс `SmartRadio` может унаследовать оба:

```python
class SmartRadio(Radio, VolumeControl):
    ...
```

Но это уже тема следующего уровня. Главное, что вы теперь знаете, как создавать и использовать **протоколы** (интерфейсы) в Python с помощью `ABC` и `@abstractmethod`.

---

## Итог по занятию

1. **Протокол** – это набор методов, которые должен поддерживать объект.
2. В Python протоколы реализуются через **абстрактные базовые классы (ABC)**.
3. Абстрактные методы помечаются `@abstractmethod`.
4. Класс, содержащий абстрактные методы, нельзя инстанциировать.
5. Наследники **обязаны** переопределить все абстрактные методы, иначе они тоже будут абстрактными.
6. Интерфейсы позволяют писать гибкий, переиспользуемый код (пример с `PowerSwitch` и разными устройствами).
7. Можно создавать не только абстрактные методы, но и конкретные методы, которые используют абстрактные (как `is_turned_off`).
8. `property` помогает сделать интерфейс более естественным.

Теперь вы готовы применять этот подход в своих проектах. Главное – не злоупотреблять абстракциями там, где они не нужны, но когда требуется чёткий контракт – абстрактные классы незаменимы.
