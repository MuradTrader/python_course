## 1. Зачем нужен Git?

Git — это **децентрализованная система контроля версий**. Она позволяет сохранять «снимки» (snapshots) вашего кода в разные моменты времени. Эти снимки называются **коммитами** (commits). В отличие от централизованных систем (например, Subversion), у каждого разработчика на компьютере хранится полная история проекта.

**Что делает Git?**
- Сравнивает строчки в текстовых файлах.
- Записывает, что было добавлено (зеленым) и что удалено (красным).
- Позволяет вернуться к любому сохраненному состоянию.
- Позволяет работать с **ветками** (branches) — параллельными линиями разработки.

**Когда делать коммит?**
- Когда вы закончили небольшой логический этап (например, добавили функцию).
- Не надо коммитить каждую опечатку, но и не надо ждать, пока напишете 500 строк.
- Лучше делать много маленьких коммитов, чем один огромный.

## 2. Начало работы. Инициализация репозитория.

Допустим, у вас есть проект на Python с файлом `main.py`:

```python
# main.py
def main():
    print("hello main")

if __name__ == "__main__":
    main()
```

Чтобы Git начал отслеживать изменения, нужно инициализировать репозиторий в папке проекта.

**Команда в терминале:**
```bash
git init
```

**Вывод в консоли:**
```
Initialized empty Git repository in /home/user/my_project/.git/
```

Git создает скрытую папку `.git`, где будет хранить всю историю. Если выполнить `ls -la`, вы увидите эту папку (она начинается с точки).

## 3. Что не нужно версионировать? Файл `.gitignore`

В проекте есть файлы, которые генерируются автоматически или являются временными:
- Виртуальное окружение (`venv/`, `.venv/`)
- Кэш Python (`__pycache__/`, `*.pyc`)
- Настройки IDE (`.idea/` для PyCharm, `.vscode/`)
- Скомпилированные файлы и бинарники

Git не должен их отслеживать. Для этого создается файл **`.gitignore`** в корне проекта.

Содержимое типичного `.gitignore` для Python (сокращенный вариант из видео):
```
# IDE
.idea/

# Virtual environment
venv/
env/

# Python cache
__pycache__/
*.pyc

# Build artifacts
build/
dist/
*.egg-info/
```

**Как это работает:** Git читает правила из `.gitignore` и не показывает эти файлы в `git status`, не добавляет их в коммиты.

После создания `.gitignore` выполните:
```bash
git status
```

**Вывод (пример):**
```
On branch master

No commits yet

Untracked files:
  (use "git add <file>..." to include in what will be committed)
        .gitignore
        main.py

nothing added to commit but untracked files present (use "git add" to track)
```

## 4. Первый коммит (initial commit)

Сначала нужно **добавить** файлы в «индекс» (staging area) — область, где собираются изменения перед коммитом.

```bash
git add .gitignore main.py
```

Теперь выполним коммит с сообщением:
```bash
git commit -m "initial commit: add .gitignore and main.py"
```

**Вывод:**
```
[master (root-commit) a1b2c3d] initial commit: add .gitignore and main.py
 2 files changed, 30 insertions(+)
 create mode 100644 .gitignore
 create mode 100644 main.py
```

Каждый коммит имеет уникальный хеш (например, `a1b2c3d`). Сообщение (`-m`) должно быть осмысленным.

```bash
git log --oneline
```

**Вывод:**

## 6. Ветки (branches)

Ветка — это отдельная линия разработки. Основная ветка обычно называется `master` или `main`. Вы можете создать новую ветку для новой функциональности (feature branch), чтобы не ломать стабильный код.

**Создать новую ветку:**
```bash
git branch feature/demo_zip
```

**Переключиться на неё:**
```bash
git checkout feature/demo_zip
```

**Создать и сразу переключиться:**
```bash
git checkout -b feature/demo_map
```

Теперь вы работаете в новой ветке. Все коммиты будут добавляться только в неё.

## 7. Пример работы в ветке: функция с zip

Допустим, в ветке `feature/demo_zip` мы пишем функцию, демонстрирующую работу встроенной функции `zip()`. Изменим `main.py`:

```python
def demo_zip():
    values1 = list(range(2, 10))   # [2,3,4,5,6,7,8,9]
    values2 = list(range(10, 29, 3)) # [10,13,16,19,22,25,28]

    for v1, v2 in zip(values1, values2):
        print(f"v1={v1}, v2={v2}, sum={v1+v2}")

def main():
    # print("hello main")   # закомментировали
    demo_zip()

if __name__ == "__main__":
    main()
```

**Что выведет этот код?** Давайте выполним его в консоли (представим, что мы запустили `python main.py`):

```
v1=2, v2=10, sum=12
v1=3, v2=13, sum=16
v1=4, v2=16, sum=20
v1=5, v2=19, sum=24
v1=6, v2=22, sum=28
v1=7, v2=25, sum=32
v1=8, v2=28, sum=36
```

**Объяснение:** `zip` объединяет элементы из нескольких итераторов до тех пор, пока не закончится самый короткий. Здесь `values1` имеет 8 элементов (2..9), `values2` — 7 элементов (10,13,16,19,22,25,28). Поэтому будет 7 итераций. Элемент 9 из первого списка не будет использован.

**Коммитим изменения:**
```bash
git add main.py
git commit -m "demo_zip: пример с zip"
```

Теперь ветка `feature/demo_zip` содержит этот коммит, а ветка `master` — нет.

## 8. Переключение между ветками и cherry-pick

Вернемся в `master`:
```bash
git checkout master
```

В `master` наш `main.py` всё еще имеет старый вид (без `demo_zip`).

Теперь предположим, что в `master` мы изменили кавычки в `print("hello main")` на двойные. Сделаем коммит в `master`:
```bash
git add main.py
git commit -m "update quotes: use double quotes"
```

Теперь хотим применить это изменение (и только его) в ветке `feature/demo_zip`. Для этого используется **cherry-pick** — «выбрать вишенку».

```bash
git checkout feature/demo_zip
git cherry-pick master   # можно указать хеш коммита или имя ветки
```

Git возьмет изменения из последнего коммита ветки `master` и применит их поверх текущей ветки. Будет создан новый коммит с таким же сообщением и автором, но с новым хешем.

**Результат:** теперь в ветке `feature/demo_zip` есть и функция `demo_zip`, и изменение кавычек.

---

## 9. Слияние веток (merge)

Допустим, мы закончили разработку `demo_zip` и хотим добавить эту функциональность в основную ветку `master`.

```bash
git checkout master
git merge feature/demo_zip
```

Если конфликтов нет, Git создаст **merge commit** — специальный коммит с двумя родителями. В истории появится запись:
```
Merge branch 'feature/demo_zip' into master
```

**Что произойдет с файлами?** Все изменения из ветки `feature/demo_zip` (коммит с `demo_zip` и `cherry-pick` кавычек) теперь появятся в `master`.

## 10. Разрешение конфликтов (conflicts)

Конфликт возникает, когда в двух ветках изменены одни и те же строки файла. Например, в `master` мы добавили функцию `demo_map`, а в `feature/demo_zip` тоже что-то меняли в том же месте.

При попытке слить `feature/demo_map` в `master` Git сообщит:
```
Auto-merging main.py
CONFLICT (content): Merge conflict in main.py
Automatic merge failed; fix conflicts and then commit the result.
```

В файле `main.py` появятся маркеры:
```python
<<<<<<< HEAD
    demo_map()
=======
    demo_zip()
>>>>>>> feature/demo_map
```

Вы должны вручную оставить нужный код (или объединить оба вызова) и убрать маркеры. Затем:
```bash
git add main.py
git commit -m "merge demo_map and resolve conflicts"
```

## 11. Rebase (перебазирование)

**Rebase** — это способ перенести коммиты из одной ветки поверх другой, создавая линейную историю. В отличие от merge, rebase не создает отдельный merge commit.

Пример: вы находитесь в ветке `feature/demo_map`, которая ответвилась от `master` давно. В `master` появились новые коммиты. Вы хотите обновить свою ветку, как будто вы начали работу от последнего состояния `master`.

```bash
git checkout feature/demo_map
git rebase master
```

Git возьмет коммиты, которые есть в `feature/demo_map`, но отсутствуют в `master`, и по очереди применит их поверх `master`. Если возникают конфликты, их нужно решать на каждом шаге.

**Преимущество:** история становится линейной, легче читается.
**Недостаток:** переписывается история (нельзя делать rebase с ветками, которые уже опубликованы в общем репозитории).

В видео показан сложный rebase с конфликтами, но суть та же: rebase переносит ваши изменения на новую базу.

## 12. GitHub и удаленный репозиторий

GitHub — это хостинг для Git-репозиториев. Он позволяет:
- Делиться кодом с другими.
- Делать резервные копии.
- Совместно работать через pull requests.

### Добавление удаленного репозитория (remote)

Создайте пустой репозиторий на GitHub (без README, .gitignore). Скопируйте URL (например, `https://github.com/yourname/demo.git`).

Локально добавьте remote:
```bash
git remote add origin https://github.com/yourname/demo.git
```

Проверить:
```bash
git remote -v
```
Вывод:
```
origin  https://github.com/yourname/demo.git (fetch)
origin  https://github.com/yourname/demo.git (push)
```

### Отправка изменений (push)

```bash
git push -u origin master
```

- `-u` устанавливает upstream (связь) между локальной веткой `master` и удаленной `origin/master`. В следующий раз можно будет писать просто `git push`.

При первом push может потребоваться авторизация (логин/пароль или токен). После успешного push ваш код появится на GitHub.

### Получение изменений (fetch / pull)

Если кто-то другой (или вы сами через веб-интерфейс) добавил коммиты на GitHub, их нужно забрать.

```bash
git fetch origin        # загружает новые данные, но не сливает с вашей веткой
git pull origin master  # fetch + merge (или rebase, если настроено)
```

В видео используется **rebase** при обновлении: `git pull --rebase`. Это более чистая история.

## 13. Восстановление потерянных изменений (reflog)

Git хранит журнал всех перемещений указателя `HEAD`. Даже если вы сделали `git reset --hard` и «потеряли» коммиты, их можно вернуть через `git reflog`.

```bash
git reflog
```

Вывод (пример):
```
a1b2c3d HEAD@{0}: reset: moving to a1b2c3d
e4f5g6h HEAD@{1}: commit: demo_map complete
i7j8k9l HEAD@{2}: commit: demo_zip another example
...
```

Выбираем нужный хеш и делаем `git reset --hard <хеш>` — восстанавливаем состояние.

**Важно:** `git reflog` хранит записи около 90 дней (по умолчанию), но не вечен.

## 14. Пример с map (дополнительно из видео)

В ветке `feature/demo_map` был показан пример с `map`. Давайте его разберем.

```python
def demo_map():
    list1 = [1, 2, 3, 5]
    list2 = [5, 3, 2, 0]
    # возводим элемент из list1 в степень из list2: 1^5, 2^3, 3^2, 5^0
    powers = list(map(pow, list1, list2))
    print(powers)

def main():
    # demo_zip()   # закомментировали
    demo_map()
```

**Вывод:**
```
[1, 8, 9, 1]
```

Объяснение: `map(pow, list1, list2)` применяет функцию `pow(x, y)` к парам элементов из двух списков. Результат — итератор, мы превращаем его в список.

## Итоговые важные моменты

| Команда | Что делает |
|---------|-------------|
| `git init` | Создать новый репозиторий |
| `git status` | Показать состояние (измененные/неотслеживаемые файлы) |
| `git add <файл>` | Добавить файл в staging area |
| `git commit -m "сообщение"` | Сохранить snapshot |
| `git log` | Показать историю коммитов |
| `git branch <имя>` | Создать ветку |
| `git checkout <ветка>` | Переключиться на ветку |
| `git merge <ветка>` | Слить ветку в текущую |
| `git rebase <ветка>` | Перебазировать текущую ветку на другую |
| `git cherry-pick <коммит>` | Взять один коммит из другой ветки |
| `git remote add origin <url>` | Добавить удаленный репозиторий |
| `git push -u origin master` | Отправить изменения на GitHub |
| `git pull --rebase` | Забрать изменения и перебазировать |
| `git reflog` | Показать все перемещения HEAD |

**Советы:**
- Делайте коммиты часто, с осмысленными сообщениями.
- Не коммитьте виртуальное окружение, кэш, IDE-файлы.
- Используйте ветки для каждой новой фичи.
- Перед тем как сделать `push`, обязательно `pull --rebase`, чтобы избежать конфликтов.
- Не бойтесь экспериментировать — Git позволяет откатиться почти из любой ситуации.

Если какой-то шаг остался непонятным, спрашивайте — разберем его ещё детальнее, с конкретными выводами консоли.
