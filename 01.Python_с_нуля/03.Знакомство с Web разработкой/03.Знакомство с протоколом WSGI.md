## 1. Что такое WSGI?

**WSGI** (Web Server Gateway Interface) — это спецификация, описывающая, как веб-сервер (например, Nginx, Apache, Gunicorn) общается с Python-приложением.

- **Веб-сервер** принимает HTTP-запрос от клиента (браузер, Insomnia, `curl`).
- Он преобразует этот запрос в стандартизированные объекты и передаёт их **Python-приложению**.
- Python-приложение возвращает ответ (статус, заголовки, тело), а сервер отправляет его клиенту.

### Зачем нужен WSGI, если есть фреймворки?

Фреймворки (Flask, Django) используют WSGI «под капотом», но скрывают детали.
На чистом WSGI ты сам разбираешь URL, метод, параметры, заголовки, формируешь ответ. Это **учебная цель** — понять, как всё устроено внутри.

## 2. Минимальное WSGI-приложение

По спецификации WSGI, приложение — это **функция** с двумя аргументами:

```python
def application(environ, start_response):
    # environ: словарь с данными запроса (метод, путь, заголовки, и т.д.)
    # start_response: функция, которую нужно вызвать для отправки статуса и заголовков ответа
    # Возвращает: итерируемый объект (например, список или генератор) с байтовыми строками — телом ответа
    start_response('200 OK', [('Content-Type', 'text/plain')])
    return [b'Hello, World!\n']
```

### Пояснение

- `environ` — словарь, содержащий все детали HTTP-запроса: метод (`REQUEST_METHOD`), путь (`PATH_INFO`), строку запроса (`QUERY_STRING`), заголовки (префикс `HTTP_`), и многое другое.
- `start_response` — функция, которую мы **обязаны** вызвать **один раз** до того, как вернуть тело ответа. Она принимает:
  - Статус: строка вида `'200 OK'`, `'404 Not Found'` и т.д.
  - Список кортежей `(название_заголовка, значение)` — все заголовки ответа.
- Возвращаемое значение — итератор из **байтовых строк** (строки с префиксом `b`). Это тело HTTP-ответа.

## 3. Установка WSGI-сервера Gunicorn

Gunicorn («Green Unicorn») — популярный WSGI-сервер, написанный на Python. Он запускает наше приложение и принимает HTTP-запросы.

### Установка

Открой терминал (в PyCharm: `Terminal` внизу) и выполни:

```bash
pip install gunicorn
```

Если у тебя виртуальное окружение, убедись, что оно активировано.

## 4. Создаём файл с приложением

Создай файл `main.py` (или любой другой). Напишем простое приложение, которое возвращает `Hello, World!`.

```python
# main.py

def application(environ, start_response):
    start_response('200 OK', [('Content-Type', 'text/plain')])
    return [b'Hello, World!\n']
```

**Что здесь происходит:**

- Мы игнорируем `environ` (пока не используем).
- Вызываем `start_response` со статусом `200 OK` и заголовком `Content-Type: text/plain`.
- Возвращаем список с одной байтовой строкой `b'Hello, World!\n'`.

---

## 5. Запуск через Gunicorn

В терминале, находясь в папке с `main.py`, выполни команду:

```bash
gunicorn main:application
```

- `main` — имя файла (без `.py`).
- `application` — имя функции внутри файла.

**Вывод в консоли:**

```
[INFO] Starting gunicorn 21.2.0
[INFO] Listening at: http://127.0.0.1:8000
[INFO] Using worker: sync
[INFO] Booting worker with pid: 12345
```

Сервер запущен на `127.0.0.1:8000` (localhost, порт 8000).

## 7. Изменяем ответ на HTML

Сделаем так, чтобы возвращался HTML с заголовком `<h1>`.
Изменим `main.py`:

```python
# main.py

def application(environ, start_response):
    start_response('200 OK', [('Content-Type', 'text/html')])
    body = b'<h1>Hello, World!</h1>\n'
    return [body]
```

### Перезапуск Gunicorn

Gunicorn **не перезагружается автоматически** при изменении кода. Останови его:

- Нажми `Ctrl + C` в терминале с Gunicorn.
- Запусти снова: `gunicorn main:application`

**Обнови страницу в браузере** — теперь текст жирный и крупный, как заголовок.

## 8. Почему все пути ведут на одну страницу?

Попробуй перейти по адресу:
**http://127.0.0.1:8000/anything/else**

Ты всё равно увидишь тот же `Hello, World!`.
Наше приложение **игнорирует путь** — оно отвечает одинаково на любой запрос.

Это происходит, потому что мы не анализируем `environ['PATH_INFO']`. Исправим это позже.

## 9. Что такое `environ`? Разбираем словарь запроса

Добавим в код вывод `environ` в консоль сервера. Для красивого форматирования используем `pprint`.

Изменим `main.py`:

```python
from pprint import pprint

def application(environ, start_response):
    print("=== НОВЫЙ ЗАПРОС ===")
    pprint(environ)   # выводим словарь в консоль Gunicorn
    start_response('200 OK', [('Content-Type', 'text/plain')])
    return [b'Hello, World!\n']
```

**Останови Gunicorn** (Ctrl+C) и запусти снова.
Теперь при каждом запросе в терминале будет печататься огромный словарь `environ`.

### Сделаем запрос через браузер (или через терминал с помощью `httpie`)

Если у тебя установлен `httpie` (альтернатива `curl`), выполни в **новом** терминале:

```bash
http GET 127.0.0.1:8000/hello?foo=bar&baz=123
```

Либо просто обнови страницу в браузере.

В терминале с Gunicorn ты увидишь что-то вроде (сокращённо):

```
{'CONTENT_LENGTH': '',
 'CONTENT_TYPE': '',
 'HTTP_ACCEPT': 'text/html,application/xhtml+xml,...',
 'HTTP_HOST': '127.0.0.1:8000',
 'HTTP_USER_AGENT': 'Mozilla/5.0 ...',
 'PATH_INFO': '/hello',
 'QUERY_STRING': 'foo=bar&baz=123',
 'REQUEST_METHOD': 'GET',
 'SCRIPT_NAME': '',
 'SERVER_NAME': '127.0.0.1',
 'SERVER_PORT': '8000',
 'wsgi.version': (1, 0),
 ...}
```

**Ключевые поля:**

- `REQUEST_METHOD` — HTTP-метод (GET, POST, ...)
- `PATH_INFO` — путь запроса (например, `/hello`)
- `QUERY_STRING` — строка параметров после `?` (без декодирования)
- `HTTP_*` — все входящие заголовки. Например, `HTTP_USER_AGENT` — это заголовок `User-Agent`.

Таким образом, мы можем **вручную** обработать разные пути и методы.

## 10. Реализуем простую маршрутизацию

Сделаем приложение, которое:

- На `/` возвращает `Home page`.
- На `/hello` возвращает `Hello, user!`.
- На любой другой путь — `404 Not Found`.

```python
from pprint import pprint

def application(environ, start_response):
    # Получаем метод и путь
    method = environ['REQUEST_METHOD']
    path = environ['PATH_INFO']

    # Логируем для наглядности (в консоль сервера)
    print(f"Request: {method} {path}")

    # Простейшая маршрутизация
    if path == '/':
        start_response('200 OK', [('Content-Type', 'text/plain')])
        return [b'Home page\n']
    elif path == '/hello':
        start_response('200 OK', [('Content-Type', 'text/plain')])
        return [b'Hello, user!\n']
    else:
        start_response('404 Not Found', [('Content-Type', 'text/plain')])
        return [b'404 - Page not found\n']
```

**Проверим:**

- `http://127.0.0.1:8000/` → `Home page`
- `http://127.0.0.1:8000/hello` → `Hello, user!`
- `http://127.0.0.1:8000/foo` → `404 - Page not found`

**Вывод в консоли Gunicorn:**

```
Request: GET /
Request: GET /hello
Request: GET /foo
```

**Важно:** В WSGI нет встроенного парсинга `QUERY_STRING`. Если тебе нужны параметры, разбирай их вручную (например, `urllib.parse.parse_qs(environ['QUERY_STRING'])`). Это неудобно — поэтому и нужны фреймворки.

## 11. Добавляем заголовок `Content-Length`

HTTP-сервер может передавать длину тела ответа в байтах. Это необязательно, но полезно.
Вычислим длину нашего ответа и добавим заголовок.

```python
def application(environ, start_response):
    body = b'<h1>Hello with length</h1>\n'
    content_length = str(len(body))   # длина в байтах
    headers = [
        ('Content-Type', 'text/html'),
        ('Content-Length', content_length)
    ]
    start_response('200 OK', headers)
    return [body]
```

**Перезапусти Gunicorn**, сделай запрос и в инструментах разработчика браузера (вкладка Network → заголовки ответа) увидишь `Content-Length`.

## 12. Обработка POST-запросов и чтение тела

WSGI позволяет читать тело запроса из `environ['wsgi.input']`. Это файлоподобный объект.
Пример для чтения POST-данных:

```python
def application(environ, start_response):
    method = environ['REQUEST_METHOD']
    if method == 'POST':
        # Читаем тело запроса (ограничим 1 МБ)
        content_length = int(environ.get('CONTENT_LENGTH', 0))
        body_bytes = environ['wsgi.input'].read(content_length) if content_length else b''
        response_text = f"Received POST data: {body_bytes.decode()}"
        body = response_text.encode()
        start_response('200 OK', [('Content-Type', 'text/plain')])
        return [body]
    else:
        start_response('200 OK', [('Content-Type', 'text/plain')])
        return [b'Use POST method to send data']
```

**Проверка через `httpie`:**

```bash
http POST 127.0.0.1:8000 name=Alice age=30
```

Сервер выведет что-то вроде:
`Received POST data: name=Alice&age=30`

(если отправлять `--form` или `--json`, формат может отличаться).

## 13. Почему чистый WSGI не используют для реальных проектов?

Из примера видно:

- **Ручная маршрутизация** (если путей много — код становится огромным).
- **Ручной парсинг** строки запроса (`QUERY_STRING`), заголовков, тела.
- **Нет поддержки** JSON, форм, файлов, кук, сессий.
- **Нет шаблонизации** (HTML генерируем конкатенацией строк).
- **Сложная обработка ошибок** и промежуточного ПО (middleware).

Поэтому разработчики используют **фреймворки** (Flask, Django, FastAPI), которые:

- Дают удобные декораторы для маршрутов (`@app.route('/hello')`).
- Автоматически парсят параметры, JSON, форму.
- Предоставляют `request` и `response` объекты.
- Поддерживают middleware, шаблоны, базы данных.

Но **все эти фреймворки внутри используют WSGI** (или ASGI — асинхронный вариант). Понимание WSGI помогает отлаживать «подкапотные» проблемы и писать собственное middleware.
