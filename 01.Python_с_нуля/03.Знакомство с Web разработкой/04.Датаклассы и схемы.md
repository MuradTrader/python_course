## 1. Проблема: простые словари и классы

Когда мы получаем данные (из базы, из API, из формы), нам нужно хранить их в структурированном виде.
Самый простой способ — словарь:

```python
user = {"id": 1, "name": "Alice", "age": 30}
print(user["name"])  # Alice
```

**Недостатки словаря:**
- Нет автодополнения ключей (можно ошибиться: `user["nme"]`).
- Нет гарантии, что ключи вообще есть (может быть `KeyError`).
- Нет подсказок по типам.

Можно написать обычный класс:

```python
class User:
    def __init__(self, id, name, age):
        self.id = id
        self.name = name
        self.age = age

user = User(1, "Alice", 30)
print(user.name)
```

**Недостатки:** много шаблонного кода (`__init__`, `__repr__`, `__eq__` и т.д.).
Именно для автоматизации этого и нужны NamedTuple, DataClass, Pydantic.

## 2. NamedTuple — кортеж с именами полей

### 2.1 Создание и использование

NamedTuple создаёт **подкласс кортежа (tuple)**, у которого поля доступны и по индексам, и по именам.

```python
from collections import namedtuple

# Создаём класс User с полями id, name, age
User = namedtuple('User', ['id', 'name', 'age'])

# Создаём экземпляр
user1 = User(1, "Alice", 30)
user2 = User(id=2, name="Bob", age=25)  # можно именованными аргументами

print(user1)
print(user2.name)
print(user1[0])   # доступ по индексу, как в кортеже
```

**Вывод:**

```
User(id=1, name='Alice', age=30)
Bob
1
```

### 2.2 Распаковка и неизменяемость

```python
# Распаковка как в кортеже
id, name, age = user1
print(id, name, age)   # 1 Alice 30

# Неизменяемость
try:
    user1.age = 31
except AttributeError as e:
    print("Ошибка:", e)   # can't set attribute
```

**Вывод:**

```
1 Alice 30
Ошибка: can't set attribute
```

### 2.3 Методы и сравнение

NamedTuple имеет методы `_asdict()`, `_replace()` (создаёт копию с заменой полей).

```python
user3 = user1._replace(age=31)
print(user3)                     # User(id=1, name='Alice', age=31)
print(user1 == user3)            # False (разные значения)
```

### 2.4 Проблема: сравнение по содержимому, игнорируя тип

Это **самый опасный недостаток** NamedTuple. Два разных namedtuple с одинаковыми значениями считаются равными, даже если у них разные имена классов.

```python
Point = namedtuple('Point', ['x', 'y'])
UserIDAge = namedtuple('UserIDAge', ['user_id', 'age'])

p = Point(40, 50)
u = UserIDAge(40, 50)

print(p == u)   # True — сравнивается содержимое кортежа (40, 50) == (40, 50)
print(type(p), type(u))  # <class '__main__.Point'> <class '__main__.UserIDAge'>
```

**Вывод:**

```
True
<class '__main__.Point'> <class '__main__.UserIDAge'>
```

Это может привести к трудноуловимым багам.

### 2.5 Итог по NamedTuple

- **Плюсы:** лёгкий, экономит память, иммутабельный.
- **Минусы:** ведёт себя как кортеж (сравнение по содержимому, индексация), неизменяемость не всегда удобна, нет встроенной валидации.
- **Когда использовать:** для очень простых, неизменяемых структур, где важна производительность и совместимость с кодом, ожидающим кортежи. В современном Python чаще используют DataClass.

## 3. DataClass — классы для хранения данных (встроенный, Python 3.7+)

DataClass решает проблему шаблонного кода, генерируя `__init__`, `__repr__`, `__eq__`, `__hash__` и другие методы автоматически.

### 3.1 Базовый пример

```python
from dataclasses import dataclass

@dataclass
class Point:
    x: int
    y: int

p1 = Point(1, 2)
p2 = Point(x=3, y=4)
print(p1)           # Point(x=1, y=2)
print(p1 == p2)     # False (сравниваются по значениям полей)
```

**Вывод:**

```
Point(x=1, y=2)
False
```

### 3.2 Изменяемость и `frozen`

По умолчанию DataClass изменяемы. Можно заморозить:

```python
@dataclass(frozen=True)
class FrozenPoint:
    x: int
    y: int

fp = FrozenPoint(10, 20)
# fp.x = 30  # Ошибка: FrozenInstanceError
```

### 3.3 Значения по умолчанию и типы

```python
@dataclass
class User:
    id: int
    name: str = "Unknown"   # значение по умолчанию
    tags: list = None       # осторожно с изменяемыми defaults!

# Правильный способ для изменяемого default:
from dataclasses import field

@dataclass
class UserCorrect:
    id: int
    name: str = "Unknown"
    tags: list = field(default_factory=list)   # новый список для каждого экземпляра

u1 = UserCorrect(1)
u2 = UserCorrect(2)
u1.tags.append("admin")
print(u1.tags)   # ['admin']
print(u2.tags)   # []
```

**Вывод:**

```
['admin']
[]
```

### 3.4 Методы внутри DataClass

Можно добавлять любые методы, как в обычном классе.

```python
@dataclass
class Point:
    x: int
    y: int

    def move(self, dx: int, dy: int) -> None:
        self.x += dx
        self.y += dy

p = Point(0, 0)
p.move(3, 4)
print(p)   # Point(x=3, y=4)
```

### 3.5 Сравнение DataClass с NamedTuple

- DataClass — это **обычный класс**, не кортеж. Сравнение идёт по полям, а не по содержимому как у кортежа.
- DataClass изменяем (если не frozen), можно добавлять методы, наследоваться.
- DataClass генерирует `__repr__`, `__eq__`, а также опционально `__hash__` (если `frozen=True`).
- DataClass не поддерживает доступ по индексу, что обычно и нужно.

### 3.6 Недостатки DataClass

- Нет встроенной валидации типов (аннотации — это подсказки, не проверки).
- Нет преобразования типов (если передать строку вместо int — будет строка, и потом код упадёт при сложении).

Пример проблемы:

```python
@dataclass
class Point:
    x: int
    y: int

p = Point("1", "2")   # Тип не проверяется!
print(p.x + p.y)      # '12' — конкатенация строк, а не сложение чисел
```

**Вывод:** `12` (строки), а не 3.

Вот здесь на сцену выходит Pydantic.

## 4. Attrs — сторонняя библиотека (предшественник DataClass)

`attrs` появилась раньше DataClass и очень похожа на неё. Сегодня её используют реже, но в старых проектах встречается.

```python
# pip install attrs
import attr

@attr.s
class Point:
    x = attr.ib(type=int)
    y = attr.ib(type=int)

p = Point(1, 2)
print(p)   # Point(x=1, y=2)
```

`attrs` даёт валидаторы, конвертеры, неизменяемость и т.д. Однако DataClass — встроенный и покрывает большинство нужд, а Pydantic даёт больше возможностей для веб-разработки. Поэтому `attrs` мы пропустим детально.

## 5. Pydantic — валидация и преобразование типов (сторонний, но стандарт)

Pydantic — **самая мощная** библиотека для работы с данными. Она:

- Проверяет типы **во время выполнения**.
- Автоматически преобразует типы (например, строку `"123"` в число `123`, строку даты в `datetime`).
- Даёт понятные ошибки валидации.
- Идеально подходит для веб-приложений (FastAPI использует Pydantic).

### 5.1 Установка

```bash
pip install pydantic
```

### 5.2 Базовая модель

```python
from pydantic import BaseModel

class Point(BaseModel):
    x: int
    y: int

# Создаём экземпляр (только именованные аргументы!)
p = Point(x=1, y=2)
print(p)            # x=1 y=2
print(p.x, p.y)     # 1 2
print(p.dict())     # {'x': 1, 'y': 2}
```

**Вывод:**

```
x=1 y=2
1 2
{'x': 1, 'y': 2}
```

### 5.3 Преобразование типов (автоматическое)

```python
p2 = Point(x="5", y="6")   # строки превратятся в int
print(p2)                  # x=5 y=6
print(p2.x + p2.y)         # 11 (числа)
```

**Вывод:**

```
x=5 y=6
11
```

Если преобразование невозможно — Pydantic выбросит ошибку:

```python
try:
    Point(x="foo", y=2)
except Exception as e:
    print(e)
```

**Вывод (сокращённо):** `value is not a valid integer`

### 5.4 Валидация и значения по умолчанию

```python
from datetime import datetime
from typing import Optional, List

class User(BaseModel):
    id: int
    name: str = "John Doe"          # значение по умолчанию
    age: Optional[int] = None       # может быть None
    signup_date: Optional[datetime] = None
    friends_ids: List[int] = []     # осторожно: Pydantic сам создаёт новый список!
    active: bool = True

u = User(id=1, age=30)
print(u)
```

**Вывод:**

```
id=1 name='John Doe' age=30 signup_date=None friends_ids=[] active=True
```

### 5.5 Важно про изменяемые defaults

В Pydantic можно писать `friends_ids: List[int] = []` — он **автоматически** создаёт новый список для каждого экземпляра (в отличие от обычных классов и DataClass). Проверим:

```python
u1 = User(id=1)
u2 = User(id=2)
u1.friends_ids.append(100)
print(u1.friends_ids)   # [100]
print(u2.friends_ids)   # []
```

**Вывод:**

```
[100]
[]
```

### 5.6 Работа с datetime — умное преобразование

```python
u = User(id=1, signup_date="2024-12-25 14:30:00")
print(u.signup_date)                # datetime.datetime(2024, 12, 25, 14, 30)
print(type(u.signup_date))          # <class 'datetime.datetime'>
print(u.signup_date.timestamp())    # 1735126200.0
```

**Вывод (зависит от времени):**

```
2024-12-25 14:30:00
<class 'datetime.datetime'>
1735126200.0
```

### 5.7 Вложенные модели

```python
class UserProfile(BaseModel):
    user: User
    points: List[Point] = []

data = {
    "user": {"id": 10, "name": "Alice"},
    "points": [{"x": 1, "y": 2}, {"x": 3, "y": 4}]
}
profile = UserProfile(**data)
print(profile)
```

**Вывод:**

```
user=User(id=10 name='Alice' age=None signup_date=None friends_ids=[] active=True) points=[Point(x=1, y=2), Point(x=3, y=4)]
```

Pydantic автоматически создаёт экземпляры вложенных моделей из словарей.

### 5.8 Обработка ошибок валидации

```python
from pydantic import ValidationError

try:
    User(id="not an int", signup_date="never")
except ValidationError as e:
    print(e.json(indent=2))
```

**Вывод (форматированный JSON с ошибками):**

```json
[
  {
    "loc": ["id"],
    "msg": "value is not a valid integer",
    "type": "type_error.integer"
  },
  {
    "loc": ["signup_date"],
    "msg": "invalid datetime format",
    "type": "value_error.datetime"
  }
]
```

### 5.9 Кастомные валидаторы

Можно добавлять свои проверки с помощью декоратора `@validator` (в Pydantic v2 — `@field_validator`).

```python
from pydantic import BaseModel, validator

class UserWithValidator(BaseModel):
    name: str

    @validator('name')
    def name_must_contain_space(cls, v):
        if ' ' not in v:
            raise ValueError('must contain a space')
        return v

try:
    u = UserWithValidator(name="John")
except ValidationError as e:
    print(e)
```

**Вывод:** `1 validation error for UserWithValidator\nname\n  must contain a space (type=value_error)`

### 5.10 Создание модели из словаря и экспорт в словарь/JSON

```python
user_dict = {"id": 5, "name": "Sam", "age": 44, "active": False}
sam = User(**user_dict)          # распаковка словаря
print(sam.dict())                # обратно в словарь
print(sam.json())                # JSON-строка
```

**Вывод:**

```
{'id': 5, 'name': 'Sam', 'age': 44, 'signup_date': None, 'friends_ids': [], 'active': False}
{"id": 5, "name": "Sam", "age": 44, "signup_date": null, "friends_ids": [], "active": false}
```

## 6. Сравнение всех подходов

| Характеристика | NamedTuple | DataClass | Pydantic |
|----------------|------------|-----------|----------|
| Встроен в Python | Да | Да (3.7+) | Нет |
| Изменяемость | Нет (иммутабельный) | Да (можно frozen) | Да (можно frozen через `Config`) |
| Доступ по индексу | Да (как кортеж) | Нет | Нет |
| Авто-преобразование типов | Нет | Нет | **Да** |
| Валидация данных | Нет | Нет | **Да** |
| Сравнение | По содержимому кортежа | По значениям полей | По значениям полей |
| Методы (свои) | Нет (только через наследование) | Да | Да |
| Чтение из словаря/JSON | Вручную | Вручную | `parse_obj`, `from_orm` |
| Популярность в вебе | Низкая | Средняя | **Очень высокая** |

## 7. Итог и рекомендации

- **NamedTuple** — используй, если нужна простая неизменяемая структура, которая должна вести себя как кортеж (например, для ключей в словаре). В остальных случаях лучше DataClass.
- **DataClass** — идеален для внутренних структур данных в приложении, когда не нужна валидация и преобразование типов. Легковесный и встроенный.
- **attrs** — исторический интерес, сейчас практически вытеснен DataClass и Pydantic.
- **Pydantic** — **стандарт де-факто** для любых данных, поступающих извне (API, формы, файлы конфигурации). Обязателен в веб-разработке (особенно с FastAPI). Даёт валидацию, преобразование, понятные ошибки и удобную работу с JSON.

### Практический совет:

- Для скриптов и внутренних моделей без валидации → `@dataclass`.
- Для всего, что приходит из сети или от пользователя → `pydantic.BaseModel`.
